# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

This is a Docusaurus v3 documentation website that turns BlackMagic Design's raw DaVinci
Resolve API `README.txt` files into structured, searchable, **versioned** Markdown docs.
Each Resolve release ships a new `README.txt`; the maintainer's job is to diff it against
the previous release, hand-port the changes into the live Markdown docs, and cut a new doc
version.

Site is deployed to GitHub Pages at `https://wheheohu.github.io/bmd_doc/`.

## Build & Development

- Package manager: **Yarn 1.22.22** (pinned via `packageManager` + `engines.node >= 18`; CI uses Node 20).
- `yarn` — install dependencies
- `yarn start` — dev server with hot reload (opens the ResolveAPI docs)
- `yarn build` — production build into `build/`. **This is the real correctness check**: it
  parses every MDX file and validates links/anchors. Run it after any doc edit.
- `yarn serve` — serve the built `build/` locally
- `yarn typecheck` — `tsc` type check (only catches TS/config issues, not MDX/link issues)
- `yarn clear` — clear Docusaurus cache (`.docusaurus/`) when the dev server gets into a bad state

### Broken-link behavior
`onBrokenLinks` and `markdown.hooks.onBrokenMarkdownLinks` are both set to **`warn`** (not
`throw`), so a bad link will **not** fail the build — you must scan the build log for
`Broken link`/`Broken anchor` warnings yourself. Broken anchors most often come from getting
a cross-link's `#slug` wrong (see anchor-slug rules below).

### Environment note (this machine)
`node`/`yarn` are provided through an nvm lazy-loader shell function that is broken in
non-interactive shells (infinite `_lazy_load_nvm` recursion). To run the toolchain from a
script/tool, bypass the function with the real binary, e.g.:

```bash
unset -f node npm npx yarn 2>/dev/null
export PATH="$HOME/.nvm/versions/node/v20.17.0/bin:$PATH"
./node_modules/.bin/docusaurus build      # or docusaurus start / docusaurus docs:version:...
```

In a normal interactive terminal the plain `yarn <script>` commands work fine.

## Architecture

Two **independent** versioned docs instances share one Docusaurus site:

| Instance | Docs plugin | Live docs | Versioned docs | Sidebar file | Versions file | Route |
|---|---|---|---|---|---|---|
| **ResolveAPI** (main) | preset-classic `docs` (`id: ResolveAPI`) | `docs/ResolveAPI/` | `ResolveAPI_versioned_docs/` | `sidebars.ts` | `ResolveAPI_versions.json` | `/ResolveAPI` |
| **workflow** | `@docusaurus/plugin-content-docs` (`id: workflow`) | `docs/workflow/` | `workflow_versioned_docs/` | `sidebarsWorkflow.ts` | `workflow_versions.json` | `/workflow` |

- Config: `docusaurus.config.ts`. ResolveAPI is the preset's built-in docs instance; workflow
  is a second explicit plugin instance. The navbar has a version dropdown per instance.
- Both sidebars are fully **autogenerated** from the folder tree (`{type: 'autogenerated', dirName: '.'}`),
  so **you never edit sidebar files** — folder structure + `_category_.json` + frontmatter drive it.
- Local search via `@cmfcmf/docusaurus-search-local`. `cheerio` is pinned to `1.0.0-rc.12`
  (`resolutions`) for build compatibility.
- Changelog page: `src/pages/versions.md` (served at `/versions`, linked from both version dropdowns).

### ResolveAPI docs layout (`docs/ResolveAPI/`)
```
intro.md                     # landing page; holds the single "Last Updated" date
other/misc.md                # "Deprecated API" page (raw code block of removed functions)
resolve_api/                 # one .md per API object (flat method list)
  _category_.json            # label "APIs", position 2, generated-index
  Resolve.md Project.md ProjectManager.md MediaPool.md MediaPoolItem.md Folder.md
  Timeline.md TimelineItem.md MediaStorage.md Gallery.md GalleryStillAlbum.md
  Graph.md ColorGroup.md
resolve_settings/            # one .md per settings/constants "dictionary" reference
  _category_.json            # label "Settings and Properties", position 3, generated-index
  RenderSettings.md AutoCaptionSettings.md TimelineExportProperties.md ... etc.
```

### Source data
- `_README_API/<version_underscored>/README.txt` — original BMD API READMEs (e.g. `20_3_0`, `21_0_2`).
  The newest folder is the source for the current live docs.
- `_README_WORKFLOW/<version>/` — workflow README source.
- These are inputs only; they are hand-converted into the `docs/` Markdown, not auto-generated.

## README.txt → Markdown Conversion Conventions

The raw README lists methods per object like:
```
OpenPage(pageName)   --> Bool   # Switches to indicated page ...
```
Convert to Markdown following these rules **exactly** — the docs are hand-maintained and must
stay internally consistent.

### API object files (`resolve_api/<Object>.md`)
- Frontmatter only — **no H1 title** (the object name is the sidebar/nav label):
  ```markdown
  ---
  displayed_sidebar: apiSidebar
  ---
  ```
- One entry per method, in this shape (a blank line after `###` is fine; both styles exist):
  ```markdown
  ### MethodName(param1, param2)
  Return Type: `ReturnType`

  Prose description copied/cleaned from the README comment.
  ```
- Multi-line README comments (parameter lists, dict keys, examples) become the prose body,
  often as sub-bullets or a `**Parameters:**` block for newer/complex methods.

### Version markers
- New methods for a release go in a **blockquote group at the top of the file**, above the
  older groups, separated by a horizontal rule (`------` or `---`):
  ```markdown
  ---
  displayed_sidebar: apiSidebar
  ---
  > New in 21.0.2

  ### NewMethod()
  Return Type: `Bool`

  Description.

  ------

  > New in 20.3.0
  ...
  ```
- The version label matches the doc version being cut (folder `21_0_2` → version `21.0.2` →
  `> New in 21.0.2`).
- For behavior/signature changes on existing methods, use admonitions (see below) or a
  `> Change at X.Y.Z` / `> Add at X.Y.Z` blockquote.

### Admonitions & deprecations
- Status callouts use Docusaurus admonitions — only `:::warning`, `:::danger`, `:::note` are used:
  ```markdown
  :::warning

  Change at 19.1.0

  :::
  ```
- A method **moved** to another object keeps its heading, strikes through the old body with
  `~~...~~`, and adds a `:::danger` pointing to the new location:
  ```markdown
  ### ApplyGradeFromDRX(path, gradeMode, [items])

  :::danger

  Move to [Graph](./Graph.md#applygradefromdrxpath-grademode) since 19.1.0

  :::

  ~~Return Type: `Bool`~~

  ~~Old description...~~
  ```
- Fully removed/legacy functions are listed as a raw fenced block in `other/misc.md`
  ("Deprecated API"), not deleted silently.

### MDX escaping (important)
- Literal braces collide with MDX JSX. Escape them as `\{ \}` **everywhere outside inline code** —
  in `###` headings, prose, and Markdown link text:
  ```markdown
  ### RemoveMotionBlur(\{deblurOption\})
  ### GenerateSpeech(\{speechGenerationSettings\}, timecode)
  ```
- Inside backticks/code fences, braces do **not** need escaping.

### Cross-linking
- Object → object: relative link `[MediaPoolItem](./MediaPoolItem.md)`.
- Object → settings page: `[Motion Deblur Settings](../resolve_settings/MotionDeblurSettings.md)`.
- Link to a specific method anchor with `#slug`. **Anchor slugs** (github-slugger) are derived
  from the heading's rendered text by: lowercasing, **removing** `()`, `{}`, `,`, `=` and other
  punctuation, and replacing spaces with `-`. Examples:
  - `CreateSubtitlesFromAudio(autoCaptionSettings)` → `#createsubtitlesfromaudioautocaptionsettings`
  - `RemoveMotionBlur({deblurOption})` → `#removemotionblurdebluroption`
  - `GenerateSpeech({speechGenerationSettings}, timecode)` → `#generatespeechspeechgenerationsettings-timecode`
  - `TranscribeAudio(useSpeakerDetection=None)` → `#transcribeaudiousespeakerdetectionnone`
- **Settings page → API method (reverse links):** whenever a settings/reference page names an
  API method, link it back as a **code-styled link in `Object:Method` form** (colon, not dot),
  following `ProjectAndClipProperties.md` and `StudioAndAIScriptingAPIs.md`:
  ```markdown
  [`Project:GetSetting`](../resolve_api/Project.md#getsettingsettingname)
  [`MediaPoolItem:GetClipProperty`](../resolve_api/MediaPoolItem.md#getclippropertypropertynamenone)
  ```
  The link text is `` `Object:Method` `` (no params); parameter/mode detail goes in the
  surrounding prose. If the same method exists on multiple objects (e.g. `AnalyzeForSlate` on
  both `Folder` and `MediaPoolItem`), link **each object's version separately**
  (`[`Folder:AnalyzeForSlate`](…) / [`MediaPoolItem:AnalyzeForSlate`](…)`).
- Studio/AI-gated methods get a trailing line linking to the prerequisites page:
  `Refer to [Studio and AI Scripting APIs](../resolve_settings/StudioAndAIScriptingAPIs.md) for prerequisites.`

### Settings / reference pages (`resolve_settings/`)
Each is a "dictionary" or "constants" reference that methods link to. Frontmatter uses a
`title:` and `displayed_sidebar: apiSidebar`, then an intro sentence linking back to the
consuming method. **Two formatting styles, chosen by content type:**

1. **Key/value settings dicts → per-key inline backticks**, one key per line, blank-separated
   (see `RenderSettings.md`, `MotionDeblurSettings.md`, `SpeechGenerationSettings.md`):
   ```markdown
   The deblurOption setting is a dictionary containing the following keys:

   `FileName`: string

   `Codec`: string (example: "H264", "ProRes422")
   ```
2. **Enum/constant lists → a `jsx` fenced code block** (see `TimelineExportProperties.md`,
   `AutoCaptionSettings.md`, `AnalyzeSlateSettings.md`):
   ```markdown
   `markerColor` can be one of the following constants:

   ```jsx
   resolve.MARKER_BLUE
   resolve.MARKER_CYAN
   ```
   ```

Do **not** dump a whole settings dict into a bare code block, and do **not** list constants
as inline-backtick lines — match the style to the content per the two rules above.

### Other per-release touch-ups
- Bump the single date in `docs/ResolveAPI/intro.md` (`💡 Last Updated: <date>`) to the
  README's "Last Updated" line.
- Prepend a `# Resolve <version> update` section to `src/pages/versions.md`, grouping changes
  per object under `### New` / `### Changes` / `### Remove` (follow the existing entries).
- Purely cosmetic README rewordings (e.g. `Refer to "X" section` → `Refer to section "X"`) are
  **not** mirrored — the docs paraphrase.

## Cutting a New Doc Version

Do this **only after the content edits in `docs/ResolveAPI/` are reviewed**. The version cut
snapshots the current live docs:

```bash
yarn docusaurus docs:version:ResolveAPI <version>    # e.g. 21.0.2
# workflow instance:  yarn docusaurus docs:version:workflow <version>
```

After cutting the version (and before committing), regenerate the agent skill so it tracks
the new docs, then commit the regenerated `skill/` output alongside the version snapshot:

```bash
yarn build:skill      # regenerates skill/davinci-resolve-scripting/ and self-validates
git add skill/
```

The skill is generated from `docs/ResolveAPI/` by `scripts/build-skill.mjs`
(templates in `scripts/skill-templates/`). Never hand-edit files under `skill/` — edit the
docs or templates and re-run `yarn build:skill`.

This command auto-generates all three of:
1. `ResolveAPI_versioned_docs/version-<version>/` — full snapshot of `docs/ResolveAPI/`
2. `ResolveAPI_versioned_sidebars/version-<version>-sidebars.json`
3. prepends `"<version>"` to `ResolveAPI_versions.json`

No `docusaurus.config.ts` or `package.json` change is needed to add a version. By repo
convention, also `git add` the new `_README_API/<version_underscored>/README.txt` source.

## Deployment
- `yarn deploy` (requires `GIT_USER`), or `USE_SSH=true yarn deploy` — builds and pushes to GitHub Pages.

## Known Quirks
- `resolve_settings/TimelineItemPropertie s.md` has a **literal space** in its filename
  (historical typo). Leave it unless intentionally fixing, and mind the space in any link.
- `_README_API/` contains far more versions (17.x–21.x) than `ResolveAPI_versions.json`
  publishes (18.6.6–latest); not every source README was turned into a released doc version.
- Some method headings have trailing whitespace — cosmetic, harmless.
